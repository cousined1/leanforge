# Agent Brief Format

> Used when transitioning an issue to `ready-for-agent` — the brief is the
> contract the AFK agent works from without further human context.

A durable agent brief survives:
- Context loss between sessions
- Handoff to a different agent platform (Claude Code → Codex → OpenCode etc.)
- The original maintainer being unavailable
- Time decay (the brief is still useful 3 months later)

If an agent reads the brief and still has to ask the maintainer a question
to start work, the brief failed.

---

## §STRUCTURE

```markdown
> *This was generated by AI during triage.*

## Agent Brief

### Goal
{One sentence. What done looks like, in user-facing terms.}

### Context
{2-4 bullets. Why this matters. What the user / business outcome is.
Use CONTEXT.md vocabulary.}

### Relevant code
{File paths and module names that the agent must read first. Cite
`graphify-out/GRAPH_REPORT.md` god nodes if the graph exists.}

### Constraints
{Hard requirements. What MUST be true. What MUST NOT change.}

### Acceptance criteria
- [ ] Specific, testable criterion
- [ ] Specific, testable criterion
- [ ] Specific, testable criterion

### Test plan
{Which tests exist. Which to add. What "good test" looks like for this
slice (per Hard Rule #17 vertical-slice discipline).}

### Decisions already made
{Anything resolved during grilling. Reference any ADRs that constrain
this work.}

### Open questions for the agent
{If any. Each tagged with a default answer the agent should use if it
can't get clarification.}

### Out of scope
{What this slice does NOT do. Prevents scope creep.}

### Definition of done
- Acceptance criteria all checked
- Tests pass
- CONTEXT.md updated if new terms introduced
- Compound learning written to `docs/knowledge/`
- PR description references the slice
```

---

## §RULES

**Use CONTEXT.md vocabulary throughout.** If `CONTEXT.md` defines "Order
intake module", say "Order intake module" — not "the order handler" or
"OrderService". Drift is a Hard Rule #15 violation.

**Cite graph evidence when the repo has a graph.** If `graphify-out/`
exists, reference god nodes and confidence-tagged edges:

> Per `GRAPH_REPORT.md`, `AuthModule` is a god node with degree 47.
> Touching it has wide blast radius — keep changes minimal and run the
> full test suite, not just the auth tests.

**Specific, not aspirational.** Acceptance criteria like "improve
performance" are not testable. "p95 latency under 200ms on the existing
benchmark" is.

**Time-box the open questions.** Each open question gets a default action:

> **Q:** Should this update existing rows or insert new ones on conflict?
> **Default if no answer:** Insert new (preserves history). If this
> default is wrong, mark the slice HITL and request clarification before
> opening the PR.

**Write for the next agent, not for yourself.** You wrote the brief; you
won't be the one executing it. The brief is read cold.

## §EXAMPLE

```markdown
> *This was generated by AI during triage.*

## Agent Brief

### Goal
Implement partial Order cancellation so a Customer can cancel one Shipment
without cancelling the whole Order.

### Context
- Currently `POST /orders/:id/cancel` cancels the entire Order including
  all pending Shipments.
- Customers have requested partial cancel — see issue #142, 12 reports in
  the last 60 days.
- Per `CONTEXT.md`: an **Order** has many **Shipments**; an **Invoice** is
  tied to a **Fulfillment** (which is per-Shipment).

### Relevant code
- `src/ordering/cancel.ts` — current cancel logic (god node per
  `GRAPH_REPORT.md`, degree 23)
- `src/ordering/Shipment.ts` — Shipment domain object
- `src/billing/Invoice.ts` — Invoice generation tied to Fulfillment
- `tests/integration/cancel.spec.ts` — existing cancel tests

### Constraints
- MUST not regress full-Order cancel behavior (existing tests must pass)
- MUST NOT generate Invoices for cancelled Shipments
- MUST emit a `shipment.cancelled` domain event (existing event bus)

### Acceptance criteria
- [ ] `POST /orders/:id/shipments/:shipmentId/cancel` returns 200 with
      updated Shipment status
- [ ] Cancelled Shipment has status `CANCELLED`, original Order still
      has status `ACTIVE`
- [ ] No Invoice generated for the cancelled Shipment
- [ ] `shipment.cancelled` event emitted with `{orderId, shipmentId}`
- [ ] If all Shipments on an Order are cancelled, Order status flips to
      `CANCELLED` and `order.cancelled` event is emitted
- [ ] Existing full-Order cancel endpoint behavior unchanged

### Test plan
- Add integration test in `tests/integration/cancel.spec.ts`:
  - "Customer can cancel one Shipment of a multi-Shipment Order"
  - "Cancelling all Shipments cascades to Order cancellation"
  - "No Invoice is generated for cancelled Shipment"
- Vertical-slice TDD per Hard Rule #17 — RED→GREEN one test at a time.
- Test public HTTP interface only, not internal Cancel module collaborators.

### Decisions already made
- New endpoint, not a parameter to the existing one (cleaner API).
  Resolved during grilling — see thread above.
- ADR-0014 records the choice to cascade Order cancel when all Shipments
  cancel, rather than leaving an empty Order.

### Open questions for the agent
- **Q:** Should refund processing be triggered automatically when a
  Shipment is cancelled, or stay manual?
  **Default if no answer:** Stay manual (existing behavior). Open a
  follow-up ticket if automatic refund is desired.

### Out of scope
- Refund flow changes
- UI changes (separate slice)
- Notifications to the customer (separate slice)

### Definition of done
- All acceptance criteria checked
- New + existing tests passing
- CONTEXT.md unchanged (no new domain terms introduced)
- Compound learning at `docs/knowledge/partial-shipment-cancel.md`
  documenting the cascade rule
- PR description cites this brief and the cascade rule from ADR-0014
```

## §WHEN_TO_USE_HITL_INSTEAD

If any of these apply, mark the issue `ready-for-human` instead of
`ready-for-agent`:

- Acceptance criteria require aesthetic / strategic judgment
- Production access is needed
- Real-user testing is required to validate
- Open questions can't be safely defaulted
- Touching infra that can't be rolled back automatically
- Regulatory / compliance review is part of the work

The brief format is the same. The reason for HITL goes in a leading
sentence:

> **HITL because:** requires production database access to backfill
> historical Orders, which the agent does not have.
